cicdq — C++ Backend

A self-contained REST API server for the CI/CD Pipeline Maturity Questionnaire application. Built in C++17 with header-only dependencies and an in-memory data store, it exposes the same API contract as the reference Node.js backend so the frontend works against either implementation without modification.

Stack

BuildCMake 3.20+
HTTP servercpp-httplib v0.15.3
JSONnlohmann/json v3.11.3

StorageIn-memory (no DB)

Architecture

All code lives in headers under include/cicdq/. There is a single translation unit (src/main.cpp) that wires everything together. This header-only design keeps compilation simple and eliminates the need for a linker step between internal modules.

Building

Dependencies are downloaded on the first cmake configure via FetchContent — no manual installation required.

ℹ A C++17-compliant compiler is required: GCC ≥ 9, Clang ≥ 10, or MSVC 2019 (v16.0).

Linux / macOS

Shell
cmake -S . -B build
cmake --build build
# binary: ./build/cicdq_server

Windows (MSVC)

PowerShell
cmake -S . -B build
cmake --build build --config Release
# binary: .\build\Release\cicdq_server.exe

CMake targets

TargetDescription
cicdq_serverMain executable

On Windows, ws2_32 is linked automatically for Winsock support. On POSIX systems, pthread is linked for std::thread (used internally by cpp-httplib).

Configuration

Configuration is read from the process environment at startup. Copy .env.example to .env and export the variables before running (the server does not load .env files automatically).

⚠ JWT_SECRET must be changed before any non-local deployment. The built-in fallback value is not secret.
VariableDefaultDescription
PORT 3002 TCP port the HTTP server binds to on 0.0.0.0
JWT_SECRET dev_secret_change_in_prod HMAC-SHA-256 signing key. Use a cryptographically random string of at least 32 bytes in production.
FRONTEND_ORIGIN http://localhost:5173 Exact origin sent in the Access-Control-Allow-Origin response header.

Example

Shell (Linux / macOS)
export PORT=3002
export JWT_SECRET="$(openssl rand -hex 32)"
export FRONTEND_ORIGIN="http://localhost:5173"
./build/cicdq_server
PowerShell (Windows)
$env:PORT            = "3002"
$env:JWT_SECRET      = "your_strong_random_secret_here"
$env:FRONTEND_ORIGIN = "http://localhost:5173"
.\build\Release\cicdq_server.exe

Authentication

The API uses stateless JWT authentication (HS256 / HMAC-SHA-256). Tokens are issued on login or register and expire after 24 hours.

Using a token

Pass the token in the Authorization header on every protected request:

Authorization: Bearer <token>

Access levels

LevelMeaning
PublicNo token required
UserAny valid JWT
OwnerValid JWT whose id matches the resource's userId
AdminValid JWT with role == "admin"
POST /api/auth/login Public

Verifies credentials and returns a signed JWT together with the user object.

Request body

JSON
{
  "username": "admin",
  "password": "password"
}

Responses

400 Bad RequestMissing username or password
401 UnauthorizedInvalid credentials

    "id":       "a1b2c3d4-...",

POST /api/auth/register Public

Creates a new account with role "user" and returns a signed JWT.

Request body

JSON

  "username": "alice",
  "password": "secret123"

Validation

  • Both fields are required and must be non-empty
  • Password must be at least 6 characters
  • Username must not already be taken

Responses

Assessment

The assessment endpoint computes a maturity score from a set of questionnaire answers and returns the achieved level plus targeted recommendations. It is intentionally public so guests can evaluate without an account.

POST /api/assessment/evaluate Public

Request body

JSON
{
  "answers": {
    "b_cache":      "DA",
    "b_logging":    "NE",
    "b_triggers":   "Push, PR",
    "ut_terminate": "NA"
  }
}

To score against a custom questionnaire, pass categories and/or rules arrays in the same body — the server will use those instead of its current in-memory versions.

Scoring algorithm

  1. For each yes_no_na item: skip if answered "NA", add 10 to effectiveMax, add 10 to score if "DA".
  2. For each text item: add 10 to effectiveMax, add 10 to score if answer is non-empty.
  3. Normalise: normalizedScore = round(score / effectiveMax × 100), capped at 100.
  4. Assign the highest maturity level whose minScore ≤ normalizedScore.
  5. Build recommendations (capped at 15 items): upgrade tip first, then missing items.

Response

200 Response
{
  "score":     72,
  "level":     3,
  "levelName": "Definirana",
  "missing": [
    "Za dosego stopnje Kvantitativno upravljana potrebujete še 3 točk.",
    "Manjka v Build: Dependency caching",
    "Manjka v Deploy: Automated rollback"
  ]
}
StatusDescription
200 OKAssessment result
400 Bad RequestMissing or non-object answers field

Security

⚠ This backend is designed for development and demonstration. Before production use, replace SHA-256 password hashing with a dedicated password-hashing algorithm (bcrypt, scrypt, or Argon2) and enable TLS.
ConcernCurrent approachRecommendation for production
Password storage salt:SHA256(salt+password) — random 8-char hex salt, custom FIPS 180-4 implementation Replace with bcrypt (cost ≥ 12), scrypt, or Argon2id
JWT signing HMAC-SHA-256 (HS256), custom implementation (RFC 2104) Use a battle-tested JWT library; consider RS256 for multi-service setups
JWT secret Read from JWT_SECRET env var; hardcoded fallback for dev Inject via secrets manager; rotate periodically; minimum 32 random bytes
Token expiry 24 hours, checked on every request Consider shorter-lived access tokens + refresh token rotation
CORS Single allowed origin via FRONTEND_ORIGIN Keep restricted; never use * with credentialed requests
Transport Plain HTTP (cpp-httplib HTTP-only build) Terminate TLS at a reverse proxy (nginx, Caddy)
Data persistence In-memory only — all data lost on restart Attach a persistent database before any real data is stored
